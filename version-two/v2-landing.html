How We Built the Task Assistant

This isn't a generic chatbot. It was built from layered research and existing ConstructConnect data to demonstrate how Elasticsearch's AI Agent Builder can create persona-aware task assistants with minimal custom code.

What We Started With

  • ORCA tables (OOUX methodology) — defined every object, relationship, CTA, and attribute across the portal
  • Mock data specs — realistic users, roles, licenses, and access levels
  • Elasticsearch agent architecture — how ES agents work, tool definitions, persona-based tool loading
  • Platform API research — mapped agent tools to existing REST endpoints (ccs.entitlements, ccs.accessmanagement, ccs.gcipauthorization)
  • KB / knowledge base research — support articles and training content the agent can surface contextually
  • Blueprint design system — component specs so the UI matches our standards

The Three-Layer Agent Pattern

1. Watcher

Proactive alerts — the agent notices pending invitations, unused licenses, and upcoming renewals before the user asks.

2. Intelligent Search (ESRE)

The input bar is a command palette. Users type what they want to do in plain language, filtered by DLS security.

3. Conversational Agent

For multi-step tasks, the agent walks users through the workflow with contextual suggestions and inline actions.

How Personas Change the Agent

The same agent framework, loaded with different tools, creates completely different experiences. We don't build three separate UIs — we define three tool sets.

Owner

User management, licenses, billing, geographic entitlements, subscriptions, reporting

Admin

User management, licenses, geographic entitlements, reporting. No billing — the tool isn't loaded, so it redirects naturally.

Team Member

Profile, access viewing, support lookup. No user management or billing — redirects to named admins.

Why Minimal Custom Code

What developers define:

  • Tool definitions — YAML/JSON files describing each API call (the agent framework handles orchestration)
  • Persona tool sets — which tools load for which role (configuration, not code)
  • Agent UI component — one chat panel, reused everywhere
  • Watcher rules — conditions that trigger proactive alerts

What developers do NOT build:

  • Separate dashboards per role
  • Custom conditional UI logic for every permission
  • Natural language processing (Elasticsearch handles this)
  • Role-based data filtering (DLS handles this at the Lucene level)

The Key Takeaway

Tools that are not loaded cannot be called. If the billing tool isn't in the Admin's tool set, the agent has no way to access billing data. There's nothing to hide because there's nothing to call. This is security by design, not by UI.
